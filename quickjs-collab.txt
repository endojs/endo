### What we've built

We have an interactive Endo capability shell running on the
formally verified seL4 microkernel using QuickJS as the JS
engine. It boots in <1 second to a serial REPL:

  seL4 kernel → Microkit PD → QuickJS → Endo shell → endo>

The shell has pet names, eval with Compartments, capability
objects, and an interactive command loop over UART.

### QuickJS version we're using

Downloaded from: https://bellard.org/quickjs/quickjs-2024-01-13.tar.xz
Compiled files: quickjs.c, libregexp.c, libunicode.c, cutils.c, libbf.c
NOT compiled: quickjs-libc.c (we have our own libc stubs)

### How we compile QuickJS

Cross-compiled for AArch64 bare metal (seL4 Microkit PD):

  aarch64-linux-gnu-gcc \
    -mcpu=cortex-a53 -mstrict-align -nostdlib -ffreestanding \
    -O2 -DEMSCRIPTEN -D_GNU_SOURCE -DNO_CONFIG_H \
    -U_FORTIFY_SOURCE -D_FORTIFY_SOURCE=0

Linked against our own:
  - heap.c (free-list allocator over 16MB seL4 memory region)
  - libc_stubs.c (string, math stubs, printf→UART, time stubs)
  - No real libc, no POSIX, no filesystem

### MAJOR UPDATE: Real SES working on seL4! (2026-03-28)

We pulled your QuickJS-ng from ../quickjs (native-compartment
branch) and the real ses.cjs bundle. SES lockdown runs for real
on the verified kernel. Intrinsics are frozen, harden() works,
Compartment isolation works. The interactive shell boots to

**Problem: boot takes ~45 seconds.** Almost all of it is SES
walking every JS intrinsic and calling Object.freeze() on each
one. On an interpreter (no JIT), this is slow.

### REQUEST: Pre-frozen intrinsics in QuickJS (HUGE impact)

If QuickJS could start with intrinsics already frozen — baked
into the engine at the C level — lockdown() would complete in
milliseconds instead of 45 seconds. This would be the single
biggest improvement for Endo OS boot time.

What would help, in priority order:

1. **Frozen-by-default intrinsics**: Build QuickJS so that
   Object.prototype, Array.prototype, Function.prototype, etc.
   are created as non-writable/non-configurable from the start.
   SES lockdown could then skip the freeze walk entirely (or
   verify in O(1) that they're already frozen).

2. **Snapshot after lockdown**: If QuickJS supported saving a
   heap snapshot after lockdown() runs, we could load that
   snapshot on boot instead of re-running lockdown(). V8 has
   this ("startup snapshot"). Even a simple "serialize the heap
   to a blob, restore it later" would work.

3. **Native lockdown() in C**: Implement the intrinsic-freezing
   loop in C instead of JS. Walking the intrinsic graph and
   calling Object.freeze() from C would be 10-100x faster than
   doing it from interpreted JS.

4. **Lazy freezing**: Mark intrinsics as "freeze-on-first-access"
   so the cost is amortized rather than upfront.

Any of these would dramatically improve the seL4 boot experience.
Option 1 (frozen-by-default) is the most aligned with the
capability model — why should intrinsics ever be mutable?

### Boot time breakdown (both optimizations needed)

                     | No bytecode  | With bytecode (qjsc)
  No pre-freeze      | ~45s + >3m   | ~45s + ~1s = ~46s
  Pre-frozen intrins  | ~1s + >3m    | ~1s + ~1s  = ~2s

They solve different problems:
  - qjsc eliminates PARSE time (lex/parse/compile JS text)
  - pre-freeze eliminates EXECUTION time (SES freeze walk)

Both together → 2 second boot. Either alone is partial.

### What's been fixed (1 bug)

**Iterator.prototype.constructor setter** (quickjs.c line 43325)
The setter was checking `!JS_IsObject(argv[0])` — validating the
VALUE must be an object. But the spec says any value is allowed.
The check should be on `this_val` (must be an object) and should
also reject `this === Iterator.prototype` (read-only on itself).
This bug caused SES `tameFauxDataProperties` to fail with
"TypeError: not an object" during lockdown.

### What needs polyfilling (not engine bugs)

**Console methods**: QuickJS only has `console.log`. SES needs
`console.warn`, `console.error`, `console.info`, `console.debug`,
`console.trace`, `console.assert`, `console.group`, etc. These
are trivially polyfillable — just alias them to `console.log`:

    if (!console.warn) console.warn = console.log;
    if (!console.error) console.error = console.log;
    if (!console.info) console.info = console.log;
    if (!console.debug) console.debug = console.log;

This polyfill MUST run before loading the SES bundle.

### QuickJS-specific globals leaking into Compartments

`print`, `gc`, `scriptArgs` are visible inside `new Compartment()`.
This is expected — the SES Compartment shim inherits them from the
start compartment. For isolation, you'd configure the Compartment
to exclude them, or not define them in the first place (which your
bare-metal build already does by not compiling quickjs-libc.c).

### What's NOT yet implemented (non-blocking for SES)

  - `SuppressedError`, `DisposableStack`, `AsyncDisposableStack`
  - `using` / `await using` syntax
  - Intl APIs (intentionally excluded for size)

### NEW: Native Compartment API (2026-03-28)

We've implemented a native `Compartment` class in C (quickjs.c).
It creates a real isolated JSContext per compartment — not an
eval+Proxy polyfill. Features:

  - `new Compartment()` — isolated context, standard intrinsics only
  - `new Compartment({ __options__: true, globals: {...} })` — endowments
  - `compartment.evaluate(code)` — strict mode eval in isolated scope
  - `compartment.globalThis` — access to compartment's global object
  - `compartment.name` — optional debug name
  - `compartment.import(specifier)` — dynamic module import
  - Nested compartments work (child can create sub-compartments)
  - NO host globals leak (no print, gc, scriptArgs, etc.)
  - Each compartment gets its own global but shares the JSRuntime

This is available via `JS_AddIntrinsicCompartment(ctx)` for
embedders using the C API, and is included by default in
`JS_NewContext()`.

The SES polyfill Compartment still works too — after `lockdown()`,
SES replaces the native Compartment with its own hardened version
that adds the module hook API.

### DONE: Native lockdown() — 0ms instead of 45 seconds!

We implemented your #3 request (native lockdown in C) and it
completes in **<1ms** — from 45 seconds to effectively instant.

New commit on native-compartment branch: `fd15edf`

**New JS functions available:**

    lockdown()   — freeze ALL standard intrinsics in <1ms
    harden(obj)  — deep-freeze any object graph, returns obj

**New C APIs for embedders:**

    JS_FreezeIntrinsics(ctx)  — freeze all intrinsics
    JS_DeepFreeze(ctx, val)   — deep-freeze an object
    JS_AddIntrinsicLockdown(ctx) — add lockdown/harden

**How it works:**
1. Force-resolves all lazy (AUTOINIT) global properties
2. Deep-freezes every reachable object from class_proto[]
   and global properties
3. Skips the global object itself (still extensible for
   user code to define variables)
4. Handles circular references with visited-object tracking

**For your seL4 boot, you have two options:**

Option A (simplest): Call native `lockdown()` from JS before
loading SES. Then SES lockdown's freeze phase becomes a no-op
since intrinsics are already frozen.

Option B (fastest): Call `JS_FreezeIntrinsics(ctx)` from your
C initialization code (`endo_init.c`) right after creating the
JSContext. This skips JS entirely.

**Important:** The native `lockdown()` does NOT do SES's
override taming (converting data properties to accessors for
the "override mistake"). If you need override-by-assignment
after lockdown, let SES do its `enablePropertyOverrides` first,
then use native `harden()` for the final freeze step.
