# Genie Memory: Knowledge Layer (PARA)

The knowledge layer organises durable facts by entity using the PARA
framework: Projects, Areas, Resources, Archives.
It answers "what do I know about X?" — complementing the session
layer's "what happened recently?"

## Entity categories

| Category     | Definition                                  | Example                  |
|--------------|---------------------------------------------|--------------------------|
| **Projects** | Goal-oriented work with a deadline or end   | "auth middleware"        |
| **Areas**    | Ongoing responsibilities without an end     | "code style standards"   |
| **Resources**| Reference material of recurring interest    | "SES lockdown patterns"  |
| **Archives** | Completed or cold entities                  | "old migration scripts"  |

Lifecycle: entities move from Projects → Archives when completed,
or from Areas → Archives when no longer relevant.

## Directory structure

```
memory/world/
  projects/<slug>/
    summary.md
    items.md
  areas/<slug>/
    summary.md
    items.md
  resources/<slug>/
    summary.md
    items.md
  archives/<slug>/
    summary.md
    items.md
```

## Entity file format

Each entity directory contains two files.

### `summary.md`

A concise (1–3 paragraph) description of the entity: what it is,
its current status, and key decisions.
Regenerated by the reflector whenever the entity's items change.

### `items.md`

Markdown with YAML frontmatter for structured data, followed by
free-form elaborative content.

```markdown
---
id: auth-middleware
category: project
status: active
created: 2026-03-20
updated: 2026-04-01
related_entities:
  - supabase-integration
  - next-app
---

## Facts

- User chose JWT-based auth over session cookies (2026-03-22)
- Middleware must run in Edge runtime (2026-03-25)
- Due date: 2026-04-15

## Notes

Extended discussion about refresh token rotation strategy.
Decision was to use sliding-window refresh with 7-day max lifetime.
```

The YAML frontmatter supports programmatic queries (e.g. "all active
projects") while the markdown body allows arbitrarily detailed notes.

**Dependency:** `js-yaml` is needed as a dependency in genie's
package for parsing frontmatter.

## Synthesis: bridging session layer → knowledge layer

The reflector (see [session layer](genie_memory_session_layer.md))
is responsible for promoting observations into PARA entities.

**Extraction rules:**
1. Identify entities mentioned 3+ times in recent observations.
2. Create a new entity directory if one does not exist.
3. Add or update facts in `items.md`.
4. Mark superseded facts (add `superseded_by` in frontmatter or
   strikethrough in body) rather than deleting them.
5. Regenerate `summary.md` from current (non-superseded) facts.
6. Move completed projects to `archives/`.

## Cross-agent sharing

The PARA knowledge layer is the natural sharing boundary between
agents.
For now, it lives in the workspace as real files.
Eventually, entities could be published into the Endo pet-name
space so that multiple genies or other daemons can share knowledge.

## Search integration

All entity files are indexed in `memory-fts.db`.
The FTS5 index is rebuilt after each reflect cycle that modifies
entity files.
